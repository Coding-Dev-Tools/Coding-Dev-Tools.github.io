Stop committing secrets: catch .env leaks before git with a pre-commit scan

The worst part about leaking a secret in a commit isn't the moment it happens — it's that the value is now frozen in your git history. You can delete the file, rewrite history, and force-push, but clones, forks, and CI caches already have it. The only reliable defense is to never let it leave your machine. That's what a commit-hook scan is for.

Envault ships rh-envault scan, a git-secrets-style scanner that runs as a pre-commit hook and blocks commits containing leaked API keys, passwords, and tokens. It runs locally on the free tier, has no telemetry, and the installed command is rh-envault. Here's how to wire it up.

Scan your working tree before you commit

The core command scans staged and unstaged files for secrets:

# scan the current project for leaked secrets
rh-envault scan

# scan and fail the process (non-zero exit) if anything is found
rh-envault scan --fail-on-found

--fail-on-found is the mode you want in automation: it exits non-zero when a secret is detected, so whatever wraps it (a hook, a CI step, a pre-push check) stops the leak.

Wire it into a pre-commit hook

Add a hook so every commit is scanned before it's written. Create .git/hooks/pre-commit:

#!/bin/sh
# block commits that contain leaked secrets
rh-envault scan --fail-on-found

Make it executable (chmod +x .git/hooks/pre-commit), and the next time you stage a file containing an AWS key or a Stripe secret, the commit is refused with the finding printed. You fix it, unstage, and recommit — the bad value never reaches the remote.

Envault also has a hook system (pre-sync / post-sync scripts), so if you'd rather manage the hook through rh-envault itself, the scan step slots into the same lifecycle as your env sync and rotation.

Catch drift between your .env and the vault

Scanning for secrets is one half. The other is knowing when a real value is sitting in a plaintext .env instead of the encrypted vault. rh-envault diff reports drift between your .env and the AES-256-GCM encrypted vault at ~/.envault/vault.enc:

# show what's in .env that isn't in the vault (or vice versa)
rh-envault diff

# block a commit when .env has values the vault doesn't
rh-envault diff --fail-on different-secrets

Used together, scan (block leaked literals) and diff (block plaintext drift) give you two independent tripwires on every commit.

Run the same check in CI

A local hook is great until someone --no-verifys their way past it. Mirror the scan in CI so a pushed secret still gets caught:

# .github/workflows/secret-scan.yml (excerpt)
- name: Scan for leaked secrets
  run: rh-envault scan --fail-on-found
- name: Check .env drift
  run: rh-envault diff --fail-on different-secrets
